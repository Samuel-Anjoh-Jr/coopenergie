1. Blockchain Mode — What triggers demo vs. real
The original guide had an implicit condition: "if vaultAddress exists". This is now replaced with an explicit environment variable to make the intent crystal clear and allow you to run the platform without blockchain access during early development.

1.1 The BLOCKCHAIN_ENABLED flag
State
BLOCKCHAIN_ENABLED
vaultAddress
Behaviour
Full production
true
Set (deployed contract)
All actions go on-chain via the Celo relayer. Real TX hashes.
Blockchain off
false
Irrelevant
Platform runs normally. Contributions are confirmed instantly. TX hashes are deterministic placeholders for UI testing.
Contract not deployed
true
null
Platform creates the Contribution/Proposal/Vote records but skips the relay call. Logs a warning. Useful during contract deployment phase.

Add this to your .env:
  ⬤  ENV VARIABLE
BLOCKCHAIN_ENABLED=true   # set to false to disable on-chain calls during development

The check in contributions.service.ts becomes:
  ⬤  contributions.service.ts — updated condition
const blockchainEnabled = process.env.BLOCKCHAIN_ENABLED === 'true';
const vaultReady = !!cooperative.vaultAddress;
 
if (blockchainEnabled && vaultReady) {
  // Full on-chain path — calls relayerService.relayContribute()
} else if (blockchainEnabled && !vaultReady) {
  // Contract not deployed yet — confirm in DB only, log warning
  this.logger.warn(`Cooperative ${cooperativeId} has no vault address yet`);

  // Demo/dev mode — instant confirm, placeholder TX hash

    `.digest('hex').slice(0, 64)}`;`,
}
This same pattern applies to proposals, votes, and withdrawal proposals. Always check BLOCKCHAIN_ENABLED && vaultAddress before relaying.

2. Celo Wallet Setup — Step by Step
2.1 What is the relayer wallet?
The relayer wallet is a standard Ethereum-compatible wallet owned by your platform. It holds CELO (the native token) to pay gas fees on behalf of your users. Users never see it or interact with it. One wallet serves all cooperatives on the platform.

2.2 Generate the wallet — one-time setup
Run this command from anywhere in your project (viem is already in apps/backend):
  ⬤  Generate wallet — run once in terminal
# From apps/backend directory:
node -e "
  const { generatePrivateKey, privateKeyToAccount } = require('viem/accounts');
  const privateKey = generatePrivateKey();
  const account = privateKeyToAccount(privateKey);
  console.log('\nRELAYER WALLET GENERATED');
  console.log('Address:', account.address);
  console.log('Private Key:', privateKey);
  console.log('\nSave these IMMEDIATELY. Never commit the private key.');
"
Save both values before closing the terminal. The private key cannot be recovered if lost. Store it in a password manager and as a secret in Railway/GitHub Actions.

How much CELO do you need? At $0.001 per transaction and 10,000 transactions per month: ~$10/month. At 1 CELO ≈ $0.50–$0.80, you need 15–20 CELO per month for a busy platform. Start with $20 worth.

On mainnet, every action (contribution, vote, proposal, withdrawal) costs ~$0.001. For a cooperative with 20 members doing 5 actions each per month, the gas cost is $0.10/month. Fully sustainable from platform fees.

3. Fund Custody Architecture — Where the real FCFA lives
3.1 The two-layer architecture
This is the most important architectural concept. The Celo blockchain tracks ACCOUNTING. CamPay holds the actual FCFA. They are always in sync.

Layer

What it holds
Who can access
Accounting layer
Celo blockchain (CooperativeVault.sol)
A number: how much each member contributed, total accumulated, approved withdrawals

CamPay merchant account
The actual FCFA — real money
Only your platform, via CamPay API with your API key

4. Withdrawal Governance Design
4.1 Voting threshold rules
Rule
Value
Rationale
Eligible voters
Members who have ≥1 CONFIRMED contribution in this cooperative
Passive members who never contributed cannot block disbursements
Default threshold
60% of eligible voters must vote YES
Majority-plus — strong enough for trust, flexible enough to function
Minimum threshold (admin-configurable)
50%
Simple majority — lowest acceptable level
Maximum threshold (admin-configurable)
90%
Near-unanimous — highest acceptable level for very conservative cooperatives
Quorum requirement
At least 3 eligible voters must have voted (YES or NO combined) before resolution
Prevents a single member approving with 1/1 = 100%
Threshold scope
Platform admin sets global default. COOP_ADMIN can override per-cooperative within the allowed range.
Flexibility per cooperative without removing platform oversight

--- withdrawals.service.ts ---
 
Method: createWithdrawalProposal(adminUserId, cooperativeId, dto)
  dto: { amountXAF, reason, destinationType, recipientPhone?,
         recipientOperator?, recipientBankName?, recipientBankAccount?,
         recipientName }
  a. Validate: user is COOP_ADMIN
  b. Validate: cooperative.confirmedBalanceXAF >= amountXAF (cannot withdraw more than balance)
  c. Create Proposal { type: WITHDRAWAL, title: `Retrait: ${amountXAF} FCFA`, description: reason }
  d. Create WithdrawalRequest linked to the proposal
  e. Call relayerService.relayCreateProposal() on-chain
  f. Return { proposal, withdrawalRequest }
 
Method: getEligibleVoters(cooperativeId)
  Returns members who have at least 1 CONFIRMED contribution in this cooperative.
  Query: memberships JOIN contributions WHERE cooperativeId AND status=CONFIRMED
  Returns: array of { userId, amountContributed }
 
Method: evaluateWithdrawalThreshold(proposalId)
  a. Get the proposal's cooperativeId
  b. Load CooperativeSettings.withdrawalThreshold (or PlatformSettings.withdrawalThresholdDefault)
  c. Load quorumMin from PlatformSettings.withdrawalQuorumMinVotes
  d. Count eligible voters via getEligibleVoters()
  e. Count YES votes from eligible voters only:
     SELECT count(*) FROM votes WHERE proposalId AND choice=true
     AND userId IN (SELECT userId FROM eligible voters)
  f. Count NO votes from eligible voters
  g. totalEligibleVoted = yesVotes + noVotes
  h. If totalEligibleVoted < quorumMin: return { approved: false, reason: "quorum not reached" }
  i. yesPercent = (yesVotes / eligibleVoterCount) * 100
  j. If yesPercent >= threshold: return { approved: true, yesPercent }
  k. Else: check if it is mathematically impossible to reach threshold (remaining voters cannot change outcome)
     If so: return { approved: false, resolved: true }
  l. Else: return { approved: false, resolved: false } — still collecting votes
 
Method: processWithdrawalOnVote(proposalId)
  Called after every vote on a WITHDRAWAL proposal:
  a. Call evaluateWithdrawalThreshold()
  b. If approved: update Proposal to APPROVED, WithdrawalRequest to APPROVED
     Then: call disbursementService.disburse(withdrawalRequestId)
  c. If resolved but not approved: update to REJECTED
 
--- disbursement.service.ts ---
 
Method: disburse(withdrawalRequestId)
  a. Load WithdrawalRequest with cooperative
  b. Update status to DISBURSING
  c. Generate reference: `COOP-WD-${cooperativeId.slice(0,8)}-${Date.now()}`
  d. Call CamPay Transfer API:
     POST ${CAMPAY_BASE_URL}/transfer/
     Headers: Authorization: Token ${CAMPAY_TRANSFER_API_KEY}
     Body (for MoMo):
     {
       "amount": withdrawalRequest.amountXAF.toString(),
       "currency": "XAF",
       "to": withdrawalRequest.recipientPhone,
       "operator": withdrawalRequest.recipientOperator,  // "MTN" or "ORANGE"
       "description": `Retrait coopérative ${cooperative.name}`,
       "external_reference": reference
     }
  e. On CamPay success:
     - Update WithdrawalRequest: status=DISBURSED, campayReference, disbursedAt
     - Decrement cooperative.confirmedBalanceXAF by amountXAF
     - Call relayerService.relayFundsReleased() on-chain (if BLOCKCHAIN_ENABLED)
     - Write AuditLog
     - Publish GraphQL event: withdrawal.disbursed
  f. On CamPay failure:
     - Update status=FAILED, failureReason=campayError
     - Revert Proposal status to PENDING (allow retry vote)
     - Notify cooperative admin via email
 
Handle CamPay withdrawal webhook too:
POST /withdrawals/webhook — same signature validation as payment webhook
Updates withdrawal status based on CamPay async transfer confirmation.
