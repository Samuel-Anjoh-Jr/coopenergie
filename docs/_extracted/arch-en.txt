CONFIDENTIAL — Internal Architecture Reference

1. System Overview
1.1 Platform Summary
CoopEnergie is a bilingual (FR default / EN) cooperative solar funding platform for the Cameroonian market. Citizens form digital cooperatives, pool contributions, govern decisions via collective voting, and verify every financial event on the public Celo blockchain.
The architecture is built in two layers that share the same codebase and can be deployed independently:
Hackathon MVP: Next.js web app with real Celo smart contracts on Alfajores testnet, REST API calls to backend, and a polished bilingual UI.
Production: Full monorepo — web + NestJS backend + Expo mobile — with CamPay payments, offline-first sync, GraphQL queries, WebSocket subscriptions, and Celo Mainnet smart contracts.
1.2 Architecture Principles
Principle
How It Is Implemented
Blockchain-native trust
Every contribution, vote, and proposal is a real Celo transaction — independently verifiable at celoscan.io without trusting the platform.
Africa-first UX
Meta-transactions hide gas complexity; Mobile Money (CamPay) for payments; offline-first mobile for 2G environments; FR as default locale.
Write REST / Read GraphQL
Mutations use REST for simplicity and reliability. Queries use GraphQL for flexible, efficient data fetching across dashboard, ledger, and report views.
Clean replacement seams
Every external dependency (blockchain, payments, email) sits behind a service interface — replaceable without touching UI components.
Tenant isolation
Each cooperative maps to one Prisma cooperativeId scope AND one deployed CooperativeVault.sol smart contract — two layers of isolation.
Type safety everywhere
Prisma client types shared across web, backend, and mobile via packages/types. GraphQL schema generates client-side types automatically.

4. Blockchain Architecture (Celo)
4.1 Why Celo for African Cooperative Finance
Concern
How Celo Addresses It
Gas cost
Transactions cost <$0.001 USD. A cooperative with 500 actions/year spends <$0.50 in gas — fully absorbed by the platform.
User complexity
Meta-transactions via EIP-2771: the backend relayer wallet pays all gas. Users never see CELO, never install MetaMask, never manage private keys.
Mobile constraints
Celo was designed for feature phones. Its consensus is optimised for mobile validators and mobile clients.
Verification
Every action creates a real TX hash. Users click it to see the transaction on celoscan.io — independent of the CoopEnergie platform entirely.
Stability
cUSD (Celo Dollar) available for price-stable contributions alongside FCFA-denominated accounting.
Developer toolchain
Ethereum-compatible — Solidity, Hardhat, OpenZeppelin, viem, Foundry all work without modification.

4.2 Smart Contract System
Contract
Purpose
Key Functions + Events
CoopFactory.sol
Factory that deploys one CooperativeVault per cooperative. Called by the backend on cooperative creation.
deployCooperative(name, target, admin) → vaultAddress
Event: CooperativeDeployed(vaultAddress, name)
CooperativeVault.sol
Per-cooperative contract. Holds the community wallet. Records all contributions, proposals, and votes on-chain.
contribute(amountXAF)
Event: ContributionMade(member, amountXAF, blockNumber)
createProposal(title, description)
Event: ProposalCreated(id, creator)
vote(proposalId, choice)
Event: VoteCast(proposalId, voter, choice)
releaseFunds(recipient, amount, proposalId) — admin only
Event: FundsReleased(recipient, amount, proposalId)
GasRelayer.sol
Accepts EIP-2771 meta-transactions from the backend relayer wallet. Verifies the user's signature and forwards calls to the target vault.
execute(from, to, data, signature)
Validates: signature authenticity, nonce, expiry

4.3 Meta-Transaction Flow (Gas Relayer)
User submits an action (contribute / vote / propose) via web or mobile UI.
Frontend sends a REST POST request to the backend with action parameters.
Backend constructs the calldata for the relevant CooperativeVault function.
Backend signs the meta-transaction using the relayer wallet private key (stored in Railway env vars).
Backend calls GasRelayer.execute() — paying the Celo gas from the relayer wallet.
GasRelayer verifies the signature and calls CooperativeVault as if from the user's address.
Celo emits the event (ContributionMade / VoteCast / etc.) with the user's address as the actor.
Backend reads the TX hash and stores it in the Prisma database alongside the local record.
TX hash is returned to the frontend and displayed with a celoscan.io link.
The user's address appears in the on-chain event even though the backend relayer paid the gas. The contribution is permanently and publicly attributed to the correct member.

5. Hybrid API Architecture
5.1 REST + GraphQL Division of Responsibility
REST handles all writes (mutations). GraphQL handles all reads (queries + subscriptions). This is not a preference — it is an enforced architectural boundary. Write endpoints NEVER have GraphQL resolvers; read endpoints NEVER have REST controllers.

6.3 Request Lifecycle (Write via REST)
Client sends POST /contributions with { amount, cooperativeId } + Bearer JWT.
AuthGuard validates JWT; CooperativeScopeGuard verifies membership.
ContributionService creates a PENDING Contribution record in PostgreSQL.
BlockchainService.relayer calls CooperativeVault.contribute() via meta-tx → TX hash returned.
Contribution record updated: status = CONFIRMED, txHash = {hash}, blockNumber = {n}.
EventListenerService picks up ContributionMade event → writes to LedgerEvent table.
GraphQL PubSub emits onContribution subscription event to all subscribed clients.
REST response returns { contribution, txHash, celoScanUrl }.

9.1 Clean Script Behaviour
The root clean script (scripts/clean.js) removes build artifacts and terminates running processes. It does NOT remove node_modules to preserve install cache.
// scripts/clean.js
const { execSync } = require('child_process');
const { rmSync, existsSync } = require('fs');

const artifacts = [
  'apps/web/.next',
  'apps/web/out',
  'apps/backend/dist',
  'apps/mobile/.expo',
  'contracts/artifacts',
  'contracts/cache',
];

artifacts.forEach(p => {
  if (existsSync(p)) { rmSync(p, { recursive: true, force: true }); console.log(`Removed: ${p}`); }
});

// Kill running processes — NOT node_modules
try { execSync('pkill -f "bun run"', { stdio: 'ignore' }); } catch {}
try { execSync('pkill -f "next dev"', { stdio: 'ignore' }); } catch {}
try { execSync('pkill -f "nest start"', { stdio: 'ignore' }); } catch {}
console.log('Build artifacts cleared. node_modules preserved.');

11. Local Development Setup
11.1 Prerequisites
Bun >= 1.1 installed globally
Docker Desktop running (PostgreSQL 15 + Redis 7 containers)
Node.js >= 20 (for scripts that require Node)
Expo Go on a physical device for mobile development
Hardhat for contract compilation and local Celo testnet interaction
11.2 Quick Start — Web MVP Only
cd apps/web && bun install && bun dev
# Connects to Celo Alfajores testnet
# Visits: http://localhost:3000/fr (default locale: FR)
11.3 Full Monorepo Start
git clone <repo> && cd coopenergie
bun install
cp .env.example .env      # fill in all values
bun run db:up             # start Docker PostgreSQL + Redis
bun run db:migrate        # apply Prisma migrations
bun run db:generate       # generate Prisma client
bun run db:seed           # seed with Cameroonian test data
cd contracts && bun hardhat deploy --network alfajores  # deploy to testnet
bun run dev               # start web + backend
