  SECTION 1 — HOW BLOCKCHAIN HASHES ACTUALLY PROVE ACTIONS  
1. The Blockchain Proof Mechanism — Full Explanation
1.1 What a transaction hash is (and is not)
A transaction hash (TX hash) is NOT a garbled or encrypted version of your data. It is a Keccak-256 cryptographic fingerprint of the signed transaction object that was submitted to the Celo network. The hash itself is just a 64-character lookup key — think of it as a receipt number. The actual proof lives inside the transaction that hash points to.

The hash is the pointer. The event logs stored at that pointer are the proof. CeloScan reads those logs and displays them in plain human-readable form.

1.2 What Celo actually stores when an action happens
When your backend calls contribute(0xUserAddress, 850000) on the CooperativeVault smart contract, Celo permanently stores three layers of data inside that transaction:
Layer
What it contains
Who can read it
Raw transaction
The ABI-encoded function call: which contract, which function, which arguments (address and number). Immutable forever.
Anyone — it is public bytes on the chain
Transaction receipt
Gas used, success status, the block number and timestamp it was included in.
Anyone
Event logs
The decoded, human-readable events emitted by the contract. For a contribution: ContributionMade(member=0x1234..., amountXAF=850000, totalXAF=950000, timestamp=1714320000)
Anyone — CeloScan displays these as formatted tables

None of this is encrypted. All of it is public. The TX hash is the key to retrieve it. This is why celoscan.io can show you the full details when you paste a TX hash — it is simply reading what is already stored on the chain.

1.3 Step-by-step: what happens when a member contributes
  Step 1  —  Member clicks "Contribute 50,000 FCFA"
The web/mobile app sends a REST POST to your NestJS backend with { cooperativeId, amountXAF: 50000 }.
  Step 2  —  Backend creates a pending record
Contribution row created in PostgreSQL with status PENDING. No blockchain call yet.
  Step 3  —  Backend encodes the contract call
Using viem: const data = encodeFunctionData({ abi: cooperativeVaultAbi, functionName: "contribute", args: ["0xUserCeloAddress", 50000n] }). This produces an ABI-encoded byte string like 0xabcd1234... that represents "call contribute with these exact arguments".
  Step 4  —  Backend signs and submits the transaction
The relayer wallet signs a transaction: { to: vaultAddress, data: encodedCall, from: relayerWallet }. This creates a signed transaction object. Keccak-256 of this signed object = the TX hash. It is computed before submission and is deterministic.
  Step 5  —  Celo network validates and executes
Celo validators verify the relayer's signature is valid, execute the contract function, and write the state change (member's contribution amount increases). The transaction is included in a block.
  Step 6  —  CooperativeVault emits an event
Inside the contribute() function: emit ContributionMade(member, amountXAF, totalXAF, block.timestamp). Celo stores this event permanently in the transaction receipt logs. This is plain-text data — address, numbers, timestamp — readable by anyone.
  Step 7  —  Backend reads the TX hash and stores it
viem returns { txHash, blockNumber } after waiting for receipt. Backend updates the Contribution record: txHash = "0xabc...", blockNumber = 28471935, status = CONFIRMED.
  Step 8  —  User sees the proof
The UI displays the TX hash as a clickable link: https://alfajores.celoscan.io/tx/0xabc... Clicking it opens CeloScan which decodes the event logs and shows: "ContributionMade — member: 0x1234..., amountXAF: 50000, totalXAF: 950000, at block 28471935 (April 28, 2026 14:32:11 UTC)".

1.4 What makes the hash cryptographically unforgeable
Property
What it means in practice
Deterministic
The same transaction inputs always produce the same hash. If you submit the same call with the same nonce and signature, you get the same hash.
Collision-resistant
It is computationally impossible to find two different transactions that produce the same hash. Each hash uniquely identifies exactly one transaction.
Avalanche effect
Changing a single bit of the input (e.g., changing 50000 to 50001 FCFA) produces a completely different hash — not a similar one. You cannot fake a hash that decodes to slightly different data.
Signature binding
The transaction is signed with the relayer's private key. Celo validators verify the signature before accepting the transaction. A hash submitted without a valid signature is rejected.
Block finality
Once a transaction is included in a Celo block and several more blocks are added after it, reversing it would require controlling >51% of Celo validator power. For practical purposes: finality after ~5 blocks (~12 seconds on Celo).

1.6 How a judge or user verifies a specific action independently
The full verification flow, requiring zero trust in CoopEnergie:
Go to celoscan.io (or alfajores.celoscan.io for testnet).
Paste the TX hash displayed in the app into the search bar.
CeloScan displays the full transaction: the contract called (CooperativeVault address), the function called (contribute), and the arguments decoded from ABI: member address, FCFA amount.
In the "Logs" tab: see the ContributionMade event in human-readable form with exact values.
Compare the amounts and addresses with what the CoopEnergie app shows. They are identical — because the app reads these same logs from the backend's event listener, not from a separate source.
To verify the community wallet balance: search the CooperativeVault contract address. See the complete transaction history — every contribution and every withdrawal that ever happened.
This is the complete trustless verification. A member can check their own contribution, a judge can verify the platform is not fabricating data, an NGO can audit the cooperative funds — all without asking CoopEnergie for anything.

   import { initializeApp, getApps } from "firebase/app"
   import { getMessaging, getToken, onMessage } from "firebase/messaging"
 
   const firebaseConfig = {
     apiKey: process.env.NEXT_PUBLIC_FIREBASE_API_KEY,
     authDomain: process.env.NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN,
     projectId: process.env.NEXT_PUBLIC_FIREBASE_PROJECT_ID,
     storageBucket: process.env.NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET,
     messagingSenderId: process.env.NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID,
     appId: process.env.NEXT_PUBLIC_FIREBASE_APP_ID,
   }
 
   export const app = !getApps().length ? initializeApp(firebaseConfig) : getApps()[0]
 
   export async function requestNotificationPermission(): Promise<string | null> {
     if (typeof window === "undefined") return null
     if (!("Notification" in window)) return null
     const permission = await Notification.requestPermission()
     if (permission !== "granted") return null
     const messaging = getMessaging(app)
     const token = await getToken(messaging, {
       vapidKey: process.env.NEXT_PUBLIC_FIREBASE_VAPID_KEY,
       serviceWorkerRegistration: await navigator.serviceWorker.getRegistration()
     })
     return token
   }
 
   export function onForegroundMessage(callback: (payload: any) => void) {
     if (typeof window === "undefined") return () => {}
     const messaging = getMessaging(app)
     return onMessage(messaging, callback)
   }

   ]
 
3. Create apps/mobile/lib/notifications/push-token.ts:
 
   import * as Notifications from "expo-notifications"
   import * as Device from "expo-device"
   import Constants from "expo-constants"
 
   export async function registerForPushNotifications(): Promise<string | null> {
     if (!Device.isDevice) {
       console.warn("Push notifications only work on physical devices")
       return null
     }
 
     const { status: existingStatus } = await Notifications.getPermissionsAsync()
     let finalStatus = existingStatus
 
     if (existingStatus !== "granted") {
       const { status } = await Notifications.requestPermissionsAsync()
       finalStatus = status
     }
 
     if (finalStatus !== "granted") return null
 
     const projectId = Constants.expoConfig?.extra?.eas?.projectId
     const token = await Notifications.getExpoPushTokenAsync({ projectId })
     return token.data  // format: ExponentPushToken[xxxxxxxx]
   }
 
   export function configurePushHandlers() {
     // How to handle notifications while app is in foreground
     Notifications.setNotificationHandler({
       handleNotification: async () => ({
         shouldShowAlert: true,
         shouldPlaySound: true,
         shouldSetBadge: true,
       }),
     })
   }
 
4. Create apps/mobile/lib/notifications/use-push-notifications.ts:
   A hook that:
   a. Calls configurePushHandlers() once on mount
   b. Calls registerForPushNotifications()
   c. On success: calls POST /users/device-token { token, platform: "IOS" or "ANDROID" }
      Determine platform: Platform.OS === "ios" ? "IOS" : "ANDROID"
   d. Stores the token in MMKV ("push_token")
   e. Sets up notification response handler:
      Notifications.addNotificationResponseReceivedListener(response => {
        const data = response.notification.request.content.data
        if (data.type === "CONTRIBUTION") router.push("/(dashboard)/contributions")
        if (data.type === "PROPOSAL") router.push("/(dashboard)/proposals")
        if (data.type === "WITHDRAWAL") router.push("/(dashboard)/proposals")
      })
   f. Returns { pushToken, notificationsEnabled }
 
5. Call usePushNotifications() in the root dashboard layout:
   apps/mobile/app/(dashboard)/_layout.tsx
   Import and call the hook at the top level of the layout component.
 
6. On logout (in auth.ts logout function):
   const pushToken = storage.getString("push_token")
   if (pushToken) {
     await api.delete("/users/device-token", { token: pushToken })
     storage.delete("push_token")
   }
